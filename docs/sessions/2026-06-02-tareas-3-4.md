## Tarea 3 — Metadatos OG/SEO de la PWA

### Origen

Cuando un cliente comparta el enlace de su PWA por WhatsApp, redes
sociales o correo, el preview que se generaba era genérico: `<title>` =
"AppForge App" heredado del template del runtime, sin descripción ni
imagen. Rompía la promesa de marca del cliente — el QR y el link
compartido perdían toda la fuerza visual.

### Solución implementada

Tres commits bisecables (DTO → builder UI + tipos → worker injection):

- **`ab515a1`** — `feat(backend): allow description in UpdateAppConfigDto`.
  Añade `@IsOptional() @IsString() @MaxLength(200) description?: string;`
  al DTO. Sin el decorador la `ValidationPipe` global con
  `whitelist: true` lo eliminaba silenciosamente (TECH_DEBT #47).
  `updateConfig` en `apps.service.ts` ya hace shallow merge, así que un
  escalar top-level se absorbe sin más cambios.

- **`fff1c0a`** — `feat(builder): add description textarea in Identidad tab`.
  Extiende `AppConfig` y `AppManifest['appConfig']` (en runtime, type-only)
  con `description?: string`. UI en `AppIconTab.tsx`: `<textarea>` con
  `maxLength={200}` (drift imposible con el DTO), placeholder humano,
  contador `n/200`. La verificación clave del diff fue que
  `updateSection('description', value)` escribe `config.description` a
  nivel raíz (no anidado) y que `saveConfig` hace `{ smtp, ...rest } =
  config` sin allowlist, garantizando que el campo viaje en el PUT.

- **`13669bc`** — `feat(backend): inject SEO/OG meta + escapeHtml helper`.
  Step 7 de `buildPwa` reescrito: `<title>` reemplazado por regex
  (`/<title>[^<]*<\/title>/i` — robusto frente a cambios del template),
  bloque OG completo (`og:title`, `og:description`, `og:type`, `og:url`,
  `og:image`, `og:site_name`) + `<meta name="description">` + el
  `theme-color` existente. Step 5 (`webmanifest`) gana `description` (sin
  escape — `JSON.stringify` se encarga). Description sourced from
  `appConfig.description?.trim()` con fallback a `app.name` para que el
  preview nunca quede vacío.

  Helper `escapeHtml` añadido a nivel de módulo (5 caracteres: `& < > " '`,
  orden importa para no doble-escapar). Aplicado a `appName`, `description`
  y — corrección durante revisión — también a `primaryColor`. Este último
  cerraba un vector XSS pre-existente: `primaryColor` sale de
  `designTokens.colors.primary.main`, persistido vía `PUT /apps/:id/schema`
  cuyo body no tiene DTO ni validación. Un cliente podía inyectar HTML
  por API saltándose el color picker. Para un hex válido el escape es
  no-op; cierra el vector sin coste.

### Smoke validado

Curl al HTML deployed devolvió las 8 líneas esperadas (1 `<title>` + 1
`<meta description>` + 6 `og:*`) con valores correctos. Escape probado
con descripción `Test "desc" <b>` → renderiza `Test &quot;desc&quot;
&lt;b&gt;` en `og:description`. Validador OG externo confirmó preview
visual completo.


## Tarea 4 — Subdominio `apps.creatu.app` + nginx estático

### Origen y diagnóstico

El servido PWA original vivía en `api.creatu.app/pwa/{slug}/` vía dos
middlewares Express en `main.ts:80-110` (cache headers por tipo de
archivo + fallback SPA con extracción manual del slug, esquivando el
problema de path-to-regexp con wildcards). Los ficheros vivían en
`appforge-backend/public/pwa/`, acoplados al árbol de código.

Tres motivos para migrar: URL pública con jerga interna (`/pwa/`),
acoplamiento operacional (cualquier rm/rsync sobre `appforge-backend`
afectaba a las PWAs deployed), y cada asset estático pasando por el
proceso NestJS innecesariamente.

### Solución implementada

Plan gate-driven (DNS → nginx 2-fases → código → migración → smoke):

- **Gate 1 — DNS**: registro A `apps.creatu.app` → `76.13.60.85`.

- **Gate 2 — nginx en dos fases canónicas**:
  - Fase 2a: bloque `:80` con `/.well-known/acme-challenge/` →
    `/var/www/certbot` + redirect HTTPS. Cert emitido con
    `certbot --webroot` (válido hasta 2026-08-31).
  - Fase 2b: bloque `:443` con TLS, `root /var/www/apps`, locations
    regex con captura nombrada de slug. Locations replicaron las reglas
    de caché del middleware Express: `sw.js` → no-cache +
    `Service-Worker-Allowed: /{slug}/`, `webmanifest|html` → no-cache,
    assets hasheados (`.js|.css|woff2?|...|png|...`) → immutable, SPA
    fallback `try_files $uri /$slug/index.html =404`. HSTS y `nosniff`
    a nivel server.

- **Gate 3 — código** (commit `96e6c31`):
  - Nueva env `PUBLIC_PWA_URL`, patrón `process.env || fallback`
    consistente con `PUBLIC_API_URL`.
  - 12 sustituciones en `build.processor.ts`: vite `--base`,
    `webmanifest.start_url`/`scope`, `og:url`/`og:image`, `<link
    rel="manifest">`, SW register URL + scope, `pwaDir`
    (`path.join('/var/www/apps', slug)` absoluto), `pwaUrl` BD, `SCOPE`
    en `generateServiceWorker`. `apiUrl` del `app-manifest.json` (paso 3)
    explícitamente intacto — el runtime sigue llamando a `api.creatu.app`.
  - `main.ts:80-110` retirado por completo: los dos `app.use('/pwa',
    ...)` y el mkdir de `public/pwa`. Sin 301 redirect — solo `test-app`
    existía, no había enlaces de cliente en circulación que justificaran
    compatibilidad.
  - `env.production.example` documenta `PUBLIC_PWA_URL=https://apps.creatu.app`.

- **Gate 4 — migración**: `UPDATE App SET pwaUrl = 'https://apps.creatu.app/'
  || slug || '/' WHERE pwaEnabled = true;` (1 fila: `test-app`). `mv` de
  `/opt/appforge/appforge-backend/public/pwa/test-app` →
  `/var/www/apps/test-app` + `chown www-data:www-data`. Ejecutado
  inmediatamente tras el reload para no dejar ventana 404↔404.

- **Gate 5 — smoke + fix imprevisto CORS** (commit `d92e2ec`).


## Hallazgos y lecciones

### Mover un servido a otro origen rompe CORS — revisar `allowedOrigins`

Antes de Tarea 4, la PWA y la API compartían origen (`api.creatu.app`),
así que las llamadas a `/apps/.../news` eran same-origin y CORS no
intervenía. Al mover el servido a `apps.creatu.app`, las llamadas pasaron
a cross-origin. El bloque `enableCors` en `main.ts:37-40` solo tenía
`PUBLIC_BUILDER_URL` y `PUBLIC_ADMIN_URL` en `allowedOrigins`; el
navegador bloqueó toda llamada de datos. Detectado por smoke en
navegador inmediatamente tras Gate 4. Fix: una línea (`d92e2ec`)
añadiendo `process.env.PUBLIC_PWA_URL` al array. El `.filter(Boolean)`
existente preserva la seguridad cuando la env no esté definida.

**Lección**: cualquier futura migración de servido a otro dominio
implica revisar `allowedOrigins` ANTES del deploy. Listo: lo añadí como
checklist mental para próximos movimientos de origen.

### `pm2 env` y `/proc/PID/environ` pueden mentir con dotenv

Tras el reload con `--update-env`, `pm2 env $PID | grep PUBLIC_PWA_URL`
devolvió vacío y `/proc/$PID/environ` tampoco la mostraba. Falso
negativo: `@nestjs/config` (vía dotenv) inyecta variables en
`process.env` como propiedades del objeto JS de Node, NO en el
entorno del SO. La única verificación válida es el comportamiento del
código: la PWA real generada salió con URLs `https://apps.creatu.app/...`
en su HTML y BD, lo que prueba que `pwaBaseUrl` resolvió al valor
correcto en runtime.

**Lección**: no medir env vars cargadas por dotenv con herramientas del
SO; medir efectos observables.

### `nginx -t` OK no garantiza que tu bloque esté

Durante Gate 2b, los dos primeros intentos de pegar el bloque `:443` se
truncaron silenciosamente. `nginx -t` pasaba (el config no quedaba
malformado, solo incompleto) y el reload se aplicaba sin error, pero
`apps.creatu.app:443` caía al primer `:443` del archivo (el de `api`)
como default implícito, presentando el cert equivocado. Diagnóstico por
`grep -cE "server_name\s+apps\.creatu\.app"` (esperado: 2, obtenido: 1)
+ `nginx -T | grep server_name` (autoritativo, post-reload).

**Lección**: tras editar nginx, verificar la PRESENCIA del bloque
buscado con grep antes de fiarse del `nginx -t` + reload.

### Permisos de ficheros tras `copyDir`

El worker corre como root en PM2, así que los ficheros escritos a
`/var/www/apps/{slug}/` quedan `root:root` con perms 644. nginx
(corriendo como `www-data`) los lee sin problema porque el dir es 755
y los archivos son world-readable por defecto. Inocuo en operación
normal. Si el worker se ejecutara como otro usuario más adelante,
habría que ajustar.
